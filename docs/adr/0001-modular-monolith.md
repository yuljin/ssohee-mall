# ADR 0001: Start With A Modular Monolith

## Status

Accepted

## Context

키즈 드롭샵은 고객 사이트, 관리자, 결제, 주문, 배송, 알림이 필요하지만 초기 운영자는 소수이며 동시에 활성화되는 드롭도 하나다.

서비스를 미리 분리하면 배포, 인증, 트랜잭션, 관측, 장애 대응 비용이 증가한다. 특히 주문 생성, 결제 승인, 참여 수량 집계는 하나의 데이터베이스 트랜잭션 경계 안에 있는 편이 안전하다.

## Decision

- 하나의 Next.js 애플리케이션으로 고객 사이트, 관리자, API를 구현한다.
- 하나의 PostgreSQL 데이터베이스를 사용한다.
- 코드는 Drop, Catalog, Order, Payment, Reward, Fulfillment 모듈로 분리한다.
- 외부 연동은 adapter interface 뒤에 둔다.
- 모듈 간 직접 테이블 수정 대신 서비스 계층을 사용한다.

## Consequences

### Positive

- 한 번에 배포할 수 있다.
- 주문/결제 트랜잭션 구현이 단순하다.
- 초기 개발과 운영 비용이 낮다.
- 도메인 경계를 유지하면 나중에 필요한 모듈만 분리할 수 있다.

### Negative

- 코드 경계를 지키지 않으면 모듈 간 결합이 생길 수 있다.
- 트래픽이 매우 커질 경우 일부 작업을 별도 워커로 분리해야 한다.

## Extraction Triggers

다음 조건이 실제로 생길 때만 서비스를 분리한다.

- 알림 작업이 웹 요청 성능에 영향을 줌
- 대량 송장/발주 처리에 별도 워커가 필요함
- 모바일 앱과 외부 파트너가 독립 API를 요구함
- 주문량 증가로 배포와 장애 범위를 분리해야 함
